大语言模型（LLM）看起来像是在"理解"和"思考"，但它的底层只做一件事：**给定前面的文字，预测下一个词是什么**。

```
输入: "今天天气"
模型: → "真" (概率 0.3), "不" (概率 0.25), "很" (概率 0.2) ...
选择: "真"

输入: "今天天气真"
模型: → "好" (概率 0.6), "差" (概率 0.15) ...
选择: "好"
```

一个 token 一个 token 地生成，直到结束。所谓"智能"，全是这个预测能力的涌现。

本文从 token 出发，沿着推理流程逐步讲清楚 LLM 的核心机制。

## Token：模型看到的不是文字

模型不认识汉字和英文，它看到的是**数字编号**：

```
"今天天气真好" → [8345, 2156, 2156, 6721, 4532, 1897]
```

每个编号叫一个 token，大致对应一个字或半个词。模型的整个世界就是这些数字序列。

### 分词器（Tokenizer）

每个模型有自己的 **tokenizer**（分词器），它和模型绑定。训练模型之前，先在大量文本上训练一个 tokenizer，常用算法是 BPE（Byte Pair Encoding）：

```
原理：统计哪些字节/字符组合出现频率最高，反复合并

初始：每个字节是一个 token
第 1 轮：发现 "th" 出现最多 → 合并成一个 token
第 2 轮：发现 "the" 出现最多 → 合并
第 3 轮：发现 "ing" 出现最多 → 合并
...
直到词表达到预设大小（比如 10 万个 token）
```

结果就是一张固定的词表：

```
token 0    = "the"
token 1    = "ing"
token 2    = "Hello"
...
token 8345 = "今天"
```

同一句话，不同模型的切分方式可能完全不同：

```
"tokenization"

模型 A:  ["token", "ization"]           → 2 个 token
模型 B:  ["tok", "en", "ization"]       → 3 个 token
某小模型: ["t", "o", "k", "e", "n", ...] → 按字符切
```

这就是为什么"一个 token 大约 4 个英文字符"只是粗略估计——取决于具体 tokenizer 的词表。

**分词器和模型是绑定的**——因为 embedding 表（下一节介绍）是按 tokenizer 的词表训练的。换一个 tokenizer，token 编号和语义的映射关系就全乱了，模型输出会变成乱码。

## Embedding：数字变成向量

光一个编号没有语义信息。模型有一张巨大的**查找表**，把每个 token 编号映射成一个高维向量（比如 4096 维的一组浮点数）：

```
"猫"  → [0.12, -0.34, 0.78, 0.56, ...]   (4096个数)
"狗"  → [0.11, -0.31, 0.75, 0.58, ...]   (和"猫"很接近)
"汽车" → [0.92, 0.44, -0.21, 0.03, ...]  (和"猫"很远)
```

训练过程中，模型发现"猫"和"狗"经常出现在相似的上下文里（"我养了一只___"），就会把它们的向量调得越来越近。

这张 embedding 表是模型参数的一部分，训练完就固定了。它是 token 在模型里的"身份证"。

## 注意力机制：LLM 的核心

现在模型拿到了一串向量。核心问题是：**每个词在理解时，应该重点关注前面的哪些词？**

举个例子：

> "那只**猫**很可爱，我想**养它**"

"它"指的是谁？人类知道是"猫"。模型怎么知道？**靠注意力机制去发现"它"和"猫"的关联最强。**

### QKV：三个角色

每个 token 的 embedding 向量经过三个不同的矩阵变换，产生三个角色：

- **Q（Query，提问）**—— "我想知道什么"
- **K（Key，标签）**—— "我能被什么样的问题找到"
- **V（Value，内容）**—— "找到我后能获取的信息"

用图书馆找书来类比：

你走进图书馆，想找关于"Python 网络编程"的书。

- **你的需求 = Q**："我想找 Python 网络编程相关的内容"
- **每本书封面上的标签 = K**：书 A "Python 基础教程"、书 B "Java 设计模式"、书 C "Python 网络爬虫实战"、书 D "法国菜谱"
- **每本书里面的内容 = V**：书 A 讲变量和循环、书 C 讲 requests 库和 socket 编程...

过程：
```
1. 你的需求(Q) 和每本书的标签(K) 逐一比对：
   书A "Python基础"    → 有点相关  → 0.2
   书B "Java设计模式"   → 不相关   → 0.01
   书C "Python网络爬虫" → 高度相关 → 0.75
   书D "法国菜谱"      → 不相关   → 0.04

2. 按相关度去吸收每本书的内容(V)：
   结果 = 0.2×书A内容 + 0.01×书B内容 + 0.75×书C内容 + 0.04×书D内容
```

你最终获得的知识**主要来自书 C**，少量来自书 A，基本没有书 B 和书 D 的内容。

### 为什么要分成三个角色？

因为**同一个词在"被别人找"和"提供信息"时，应该展现不同的面**。

"猫"作为 K（标签）时强调的是："我是名词、可被指代"——方便代词来匹配。

"猫"作为 V（内容）时强调的是："我代表一种动物、毛茸茸的"——这才是要传递的语义信息。

如果 K 和 V 是同一个东西，就好比书的封面标签和书的内容完全一样——标签没法做到既方便检索，又承载全部内容。分工才能各自优化。

### 每个 Token 同时是提问者和被查询者

在图书馆的例子中，"你"是找书的人（Q），"书"提供标签（K）和内容（V），好像是不同的角色。但实际上，**每个 token 既是找书的人，也是书架上的一本书**。

```
"那 只 猫 很 可爱 我 想 养 它"

每个 token 都同时拥有三个身份：

"猫" → Q: "我想知道谁在修饰我"      → 用 Q 去找别人
       K: "我是名词/动物/可被指代"    → 用 K 等别人来找我
       V: "毛茸茸的四脚动物"          → 找到我后能获取的信息

"它" → Q: "我是代词，找我的指代对象"   → 用 Q 去找别人
       K: "我是代词"                  → 用 K 等别人来找我
       V: "指代关系"                  → 找到我后能获取的信息
```

所有 token 同时并行地做这两件事，一轮注意力计算就完成了。

### 从 Token 到 QKV 的具体计算

```
token "猫"
   ↓
embedding 查表 → 向量 [0.12, -0.34, 0.78, ...]  (4096维)
   ↓
   ├── × W_q 矩阵 → Q向量 [0.45, 0.21, ...]
   ├── × W_k 矩阵 → K向量 [0.33, -0.12, ...]
   └── × W_v 矩阵 → V向量 [0.67, 0.89, ...]
```

就是同一个向量乘以三个不同的矩阵，得到三个不同用途的向量。W_q、W_k、W_v 这三个矩阵就是模型训练时学到的参数。

### 注意力的完整数学过程

假设有 4 个 token `[A] [B] [C] [D]`，要计算 token D 的输出：

```
第一步：D 的 Q 向量和所有 token 的 K 向量做点积，得到相关性分数

score_A = Q_D · K_A = 2.1
score_B = Q_D · K_B = 0.3
score_C = Q_D · K_C = 5.7
score_D = Q_D · K_D = 1.2

第二步：softmax 把分数归一化成概率（加起来 = 1）

weight_A = 0.08
weight_B = 0.01
weight_C = 0.85   ← D 最关注 C
weight_D = 0.06

第三步：用这些权重对所有 V 向量加权求和

output_D = 0.08·V_A + 0.01·V_B + 0.85·V_C + 0.06·V_D
```

经过这一步，D 的输出向量就主要融合了 C 的信息。

### Attention Mask

注意力计算中有一个重要机制：**mask**。通过将某些 token 的分数设为负无穷，softmax 后它们的权重变为 0，从而被完全排除在注意力计算之外。

```
score_A = Q_D · K_A = 2.1
score_B = Q_D · K_B = -∞    ← 被 mask 掉
score_C = Q_D · K_C = 5.7
score_D = Q_D · K_D = 1.2

softmax 后：
weight_A = 0.09
weight_B = 0.00    ← e^(-∞) = 0，完全不参与
weight_C = 0.87
weight_D = 0.04
```

mask 最常见的用途是 **causal mask**（因果掩码）：确保每个 token 只能关注它前面的 token，不能"偷看"后面的。这是自回归语言模型的基本约束——预测下一个词时，不能看到未来的内容。

## 多层叠加

以上描述的是**一层**注意力。实际模型有几十层（大模型通常 80+ 层），逐层堆叠：

```
第 1 层：捕捉简单的局部关系（"它" → "猫"）
第 2 层：在第 1 层结果上，捕捉更复杂的模式
...
第 40 层：理解抽象概念、推理链路
...
第 80 层：综合所有信息，形成最终判断
```

每一层都有独立的 W_q、W_k、W_v 矩阵。同一个 token 在不同层会产生不同的 QKV 向量，因为输入已经融合了上一层的上下文信息：

```
"我 打开 窗户"  → "打开"的高层 K 偏向"物理动作"
"我 打开 浏览器" → "打开"的高层 K 偏向"启动软件"
```

模型参数量说的"200B 参数"，主要就是所有层的 W_q、W_k、W_v 矩阵加上 embedding 表的总元素个数。

## 预测输出

最后一层输出的向量，经过一个线性层映射到词表大小（比如 10 万个 token），得到每个 token 的概率分布：

```
最终向量 → [0.001, 0.003, ..., 0.15, ..., 0.002]
                                 ↑
                               "好"这个 token 概率最高
→ 输出"好"
```

然后把"好"追加到输入末尾，继续预测下一个 token，如此循环直到生成结束标记。

## KV Cache：避免重复计算

理解了以上流程后，就能明白一个关键优化。

当模型生成完"真"之后要预测下一个词时，输入变成"今天天气真"。注意力机制中，"今天天气"这四个 token 的 K 和 V **和上一轮完全一样**——因为注意力只看前面的 token，不看后面的，所以新增的"真"不会影响前面 token 的 KV。

```
已有: [今] [天] [天] [气]    ← 这些的 KV 上一步已经算过了
新增: [真]                   ← 只需要算"真"的 QKV
                               用"真"的 Q 去匹配前面所有的 K
                               从匹配到的 V 中吸收信息
```

把旧 token 的 K 和 V 缓存起来，新 token 只需要算自己的 QKV，就能完成推理。这就是 **KV Cache**。

注意：**只缓存 K 和 V，不缓存 Q**。因为旧 token 的 Q 不会再被任何人用到（没有"未来 token 的 K 去匹配旧 token 的 Q"这种操作），而旧 token 的 K 和 V 会被每一个新 token 反复使用。

KV Cache 的代价是显存占用。每一层、每一个 token 都有对应的 K 和 V 向量，200K token × 80 层的 KV Cache 可能占几个 GB 的显存。这就是模型上下文长度的物理瓶颈之一。

## 上下文窗口

所谓模型"支持 200K 上下文"，指的是单次推理能接受的最大输入 token 数。超过了，API 直接返回错误。

这个限制来自两个硬约束：

1. **显存**：KV Cache 大小和序列长度成正比，200K 的 KV Cache 已经占几 GB 显存
2. **位置编码**：模型训练时使用的位置编码有范围上限，超出训练范围的位置，模型"没见过"，输出质量会急剧下降

## 完整流程回顾

```
用户输入 "今天天气"
    ↓
分词（Tokenizer）: [8345, 2156, 2156, 6721]
    ↓
Embedding 查表: 4 个 token → 4 个向量
    ↓
第 1 层注意力: 每个向量通过 QKV 和其他向量交互，更新自己
第 2 层注意力: 在上一层输出基础上，同样的 QKV 交互
...
第 80 层注意力: 同上
    ↓
最终向量 → 线性层 → softmax → "真" 概率最高
    ↓
输出 "真"，追加到输入末尾，缓存 KV
    ↓
重复：输入变成 "今天天气真"，新 token 复用旧 KV，预测下一个...
```

以上就是 LLM 推理的完整链路。下一篇文章将介绍 Claude Code 如何基于这些机制（特别是 KV Cache）设计了多层上下文管理策略，在长对话中平衡成本、速度和信息保留。
